SLogo : Design Extensions

The changes below are intended to stretch your design further and challenge some of the assumptions made in your initial version. Focus on implementing these changes in such a way as to promote good design in your program, not simply as special cases added on without a clear plan. While creating your code for this project, we expect you to follow this design checklist before pushing your code to Gitlab.

Specifically the amount of credit you will receive for these features will be in proportion to how clearly it follows the goals of your design. Contorting your main design to add in extra functionality or adding poor code that diminishes the rest of your refactoring efforts will not be rewarded. Thus, a program with fewer features, but plenty of clear paths to easy expansion, is worth more than a program with lots of features but no closed classes and no good inheritance hierarchies. In short, to maximize your grade, implement enough variety in your program to clearly demonstrate that your design supports further such extensions.

Basic Extensions

In the early days, Logo was run on small, slow machines and thus its environments were given only limited capabilities. There was only a single turtle drawn as a triangle in only one display window to show the turtle's actions. In modern times, Logo has been used to run simulations with thousands of turtles and also to make animations. Your job in extending the basic program is to bring your SLogo environment from the seventies into this millennium.

Front end

In general, your environment should provide a variety of ways to interact with the turtle and its attributes and programs the user has written, such as entering text commands, reading commands/preferences from a file, using buttons or more complex GUI components. Also, since there are so many possible views in this version, users should be able to open or close any view (except the main turtle display).

Specifically, allow users

  • to see the state of the turtle (i.e., its position, heading, pen up/down, or other parameters affected by the environment)
  • to see palettes of images/colors with numbers (indexes) that can be referred to within new SLogo commands (i.e., it should be clear which images/colors are associated with which numbers so the SLogo language can refer to them in code). At the start, there should be a small default set of values assigned to numbers to choose from.
  • to set, save, and load workspace preferences (like default background, starting image list, starting number of turtles, starting file to load, command language, etc.)
  • to create multiple workspaces (perhaps as tabs or as separate windows). You can define exactly what a workspace contains but, at the very least, it should have separate turtle display areas, command histories, active variables, and preferences
  • to select active turtles graphically and toggle whether or not to graphically identify the currently active turtles in a workspace's display window
  • to move the current turtle(s) graphically (at least FD, BK, LT, and RT).
    Thus there should be a number of different ways to move the turtle in your IDE: interactively entering commands, using the GUI, and reading commands from a file.
  • to change the current pen's properties graphically (up/down, thickness, etc.).
    Thus there should be a number of different ways to change the pen in your IDE: interactively entering commands, using the GUI, and reading commands from a file.
Back end

Interpret an extended set of SLogo commands:

  • recognize these additional commands (in multiple languages)
  • allow users to save (and later read) a file of valid SLogo functions/variables
    Thus the user can develop a library of work which, when loaded, are added to the set of valid SLogo commands that can be entered interactively.

Challenging Extensions

Front end
  • animation: animate the turtles' movement (i.e., so users can see the turtle moving forward as it executes FD). Users should be able to control how fast the turtle animates when it performs its actions (at some point, it should be so fast that animation is not done) and to start, pause, reset, and single step the turtles' performance of commands (even better, highlight commands as they are being run).
  • undo: allow the user to undo/redo the last action(s) done in a workspace (each workspace should have a separate set of commands)
  • debugger: by stepping into/over commands, users can "debug" their programs (perhaps a small step if you have implemented the animation extension above). Even better, allow users to change the values of variables while the they are running the program (or even the code that is currently running!).
Back end
  • grouping: allow variable number of parameters to procedures where appropriate by using parentheses (e.g., sum, difference, product, and, or, etc.)
  • scope: allow dynamic variable scope and local variables
  • recursion: allow user defined commands to include recursive calls
